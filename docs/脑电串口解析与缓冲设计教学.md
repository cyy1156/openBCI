# Qt 脑电（ThinkGear / 思知金牛座风格）串口解析 — 教学演示文档

本文面向**已能看懂官方协议与解析状态机**的读者，把三件事串成一条可落地的工程路径：

1. **协议帧**：双 `0xAA` 同步头、长度、`payload`、校验和，以及 `RAW` / `EEG` 等代码字解析。  
2. **`QSerialPort` 串口层**：打开、参数、信号槽、错误与重连要点。  
3. **串口接收缓冲**：应对「每秒约 **512×8 字节** 的高频小包」+「每秒约 **1×36 字节** 的低频大包」的**粘包/半包**与**背压**问题。

> 说明：文中的 **512×8**、**1×36** 按思知/模块常见规格取为**量级估算**（用于定缓冲容量与 UI 刷新策略）。单帧总长恒为 **`4 + N`** 字节（`N = PayloadLength`）。模块侧常见：**波特率 57600**、**RAW 约 512 Hz**、**ADC 12 位**（与 RAW 的 16 位打包传输不矛盾）。

## 〇、思知（Sichiray）类模块：小包与大包（与工程代码对齐）

以下与思知蓝牙/串口教程、`EEG V6.1` 类模块说明一致，并与本仓库 `ThinkGearFrameAssembler` + `ThinkGearPayloadParser` 对应。

### 0.1 小包（高频 RAW，约 512 包/秒）

| 字段 | 示例（十六进制） | 说明 |
|------|------------------|------|
| 同步 | `AA AA` | 帧头 |
| 长度 | `04` | **Payload 仅 4 字节**（不含长度、不含校验） |
| Payload | `80 02 xxHigh xxLow` | `80` = RAW 类型；`02` 为固定第二字节（与教程一致）；原始值 = `(xxHigh<<8)|xxLow`，再按**有符号 16 位**解释（见 0.3） |
| 校验 | `xxCheckSum` | 对 **整个 Payload 共 4 字节** 求和，按位取反，取低 8 位 |

等价校验公式（教程写法）：

\[
\texttt{chk} = \bigl(\sim(\texttt{0x80}+\texttt{0x02}+\texttt{xxHigh}+\texttt{xxLow})\bigr) \,\&\, \texttt{0xFF}
\]

（即 32 位按位非后取低字节，与「先加 Payload 内各字节再 `~sum & 0xFF`」一致。）

**工程实现**：`ThinkGearFrameAssembler::checksumOk` 对 **全部 `N` 个 Payload 字节**循环累加，再 `expected = (~sum) & 0xFF`，与小包 `N=4` 时上式相同。校验失败应整帧丢弃；教程称丢包率 &lt;10% 通常可接受。

### 0.2 有符号 RAW（与小包解析一致）

```text
rawdata = (xxHigh << 8) | xxLow;
if (rawdata > 32767)   // 教程常写成 >32768，易笔误；应对 32768..65535 转负
    rawdata -= 65536;
```

本仓库 `ThinkGearPayloadParser` 中 `0x80` 分支按上式实现（协议与代码一致）。

### 0.3 大包（低频，约 1 次/秒：Signal、eSense、8 段 EEG Power）

整帧仍为：`AA AA` + **`N=0x20`（十进制 32）** + **32 字节 Payload** + **1 字节校验**，总长度 **36** 字节。

Payload 内为连续 TLV（代码字 + 数据），教程示例结构：

| 顺序 | 字节 | 含义 |
|------|------|------|
| 1 | `02` `xx` | **信号 Signal**（代码 `0x02`，下一字节为值，如 `C8`） |
| 2 | `83` `18` + 24 字节 | **EEG Power**：`18h`=24，为 **8 频段 × 3 字节大端无符号** |
| 3 | `04` `xx` | **专注度 Attention**（0–100） |
| 4 | `05` `xx` | **放松度 Meditation**（0–100） |

**8 段功率解析**：每频段 3 字节，**高字节左移 16 位 | 中字节左移 8 位 | 低字节**，结果为**无符号整数**，无绝对物理单位，仅在各频段之间相对比较有意义。

> 教程中「第 513 个大包」为形象说法：**约每 512 个 RAW 小包后夹 1 次大块**，固件不必严格计数为 513；组帧只认 `AA AA` 与长度字节。

## 一、协议帧结构（与解析器对应关系）

一帧二进制布局（大端/字节序按代码与官方说明）：

```
[0xAA][0xAA][PayloadLength][Payload…][Checksum]
 同步1  同步2    1 字节        N 字节     1 字节
```

- **`PayloadLength = N`**：仅表示 `Payload` 区长度，**不包含** 两个同步字节、长度字节本身、也不包含最后的校验字节。  
- **校验**：对 `Payload` 的每个字节求和，取反后保留低 8 位，与最后一字节比较（与你提供的 `parserData` 中 `PARSER_STATE_CHKSUM` 一致）。  
- **整帧长度**：

\[
L_{\text{frame}} = 3 + N + 1 = 4 + N
\]

其中 `3` = `0xAA + 0xAA + Length`，`+1` = `Checksum`。

`Payload` 内部是多个 **「代码字 + 数据」** 的 TLV 风格片段，例如：

| 代码 | 含义 | 典型长度（代码+数据） |
|------|------|------------------------|
| `0x02` | 信号质量 | 2 |
| `0x04` | 注意力 | 2 |
| `0x05` | 冥想 | 2 |
| `0x16` | 眨眼强度 | 2 |
| `0x80` | 16 位 RAW | 3（高字节在前） |
| `0x83` | 8 段 EEG 功率 | 26 |

**教学要点**：  
- 组帧类切出的 `frame` 为「从第一个 `0xAA` 起的完整 **`4+N` 字节**」；`ThinkGearPayloadParser::parseFrame` 仅解析其中 `N` 字节 Payload。  
- 本仓库 `parsePayload` 使用 **`while (i < n)` 扫完整段 TLV**，同一帧内若将来出现多个代码字（例如扩展固件），会顺序处理；**高频小包**与**大包**通常分属不同帧，由长度 `N=4` 与 `N=32` 等区分。

## 二、流量估算与缓冲容量设计

### 2.1 按你给出的业务量级

- **高频小包**：每秒 `512 × 8 = 4096` 字节。  
- **低频大包**：每秒 `1 × 36 = 36` 字节。  

合计约 **4132 B/s**（约 **4.0 KiB/s**），换算波特率量级：  

\[
4132 \times 10 \approx 41\text{kbit/s 量级（8N1 约 10 位/字节）}
\]

57600 波特在理论上足够，但仍建议：  
- 留 **协议开销、重传、抖动、系统调度** 余量；  
- **接收端缓冲按「峰值 × 时间窗」** 取，而不是按平均值卡死。

### 2.2 推荐缓冲大小（教学用公式）

设：

- \(R\)：应用**消费**字节的平均速率（若 UI/算法跟不上，消费会变慢）；  
- \(B_{\text{in}}\)：串口**输入**峰值字节率；  
- \(T\)：希望系统在**不丢数据**前提下能扛住的「卡顿时间」（秒），例如 `0.5s ~ 2s`。

则接收环形区或 `QByteArray` 建议至少：

\[
\text{Cap} \geq B_{\text{in}} \times T + \text{最大单帧长度}
\]

**数值示例**（偏保守，适合小白工程直接采用）：

- 取 \(B_{\text{in}} \approx 8\text{KiB/s}\)（约为你的 2 倍余量）；  
- 取 \(T = 2\text{s}\)；  
- 最大单帧：协议上 `PayloadLength` 合法上限若按你代码 `170`，则 \(L_{\text{frame}} \leq 4+170=174\)。  

则：

\[
\text{Cap} \gtrsim 8 \times 1024 \times 2 + 174 \approx 16.3\text{KiB}
\]

**实践建议**：接收侧用 **`16 KiB ~ 64 KiB`** 的软上限均可；关键是**要有「上限 + 丢弃策略/告警」**，避免异常设备或线路把 `QByteArray` 撑到无限大导致内存问题。

### 2.3 与绘图（QWT）的关系

- **RAW（512Hz）**：不要每来一个点都重绘全图；用 **环形缓冲区**（你项目里 `DataBuffer` + `RawSample` 的思路）存最近 `512×k` 个点，**定时器 30~60Hz** 刷新曲线。  
- **大包（1Hz）**：注意力/冥想/8 段功率等，**仅在 `isEeg` 且字段更新时**更新柱状图或第二条曲线。  

这样串口线程/槽函数只做「收包 + 解析 + 写入结构」，**绘图在 GUI 线程节流**，避免卡顿。

## 三、`QSerialPort` 设计（类职责与生命周期）

### 3.1 推荐模块划分

| 模块 | 职责 |
|------|------|
| `SerialReceiver`（或放在 `MainWindow` 内） | 拥有 `QSerialPort`，打开/关闭/重连，读 `readyRead` |
| `RxFrameAssembler` | 维护 `QByteArray m_rxBuffer`，从流中切出完整帧 |
| `EegParser` | 调用你的 `parserData`，输出 `raw` / `eegPkt` |
| `DataBuffer`（你已有） | 高频 `RawSample` 与低频 `BigPackageSample` 分池存储 |

### 3.2 打开串口检查清单

- **端口名**：Windows `COM7`；可用 `QSerialPortInfo::availablePorts()` 枚举给用户选。  
- **波特率**：ThinkGear 常见 `57600`（以设备文档为准）。  
- **数据格式**：`8N1`（`QSerialPort::Data8`、`NoParity`、`OneStop`）。  
- **流控**：多数脑电模块为 `NoFlowControl`。  
- **`readBufferSize`**：可显式设大一些（如 `32*1024`），**减少内核与应用之间拷贝次数**（非必须，但有益）。  

### 3.3 信号槽（线程模型）

**推荐**：`QSerialPort` 与 UI 同线程，槽里 **只做 `readAll()` + 追加缓冲 + 轻量解析**；若解析很重，用 `QtConcurrent` 或工作线程，但要把结果用 **信号 + 队列连接** 回 GUI。

典型连接：

```cpp
connect(m_port, &QSerialPort::readyRead, this, &YourClass::onReadyRead);
connect(m_port, &QSerialPort::errorOccurred, this, &YourClass::onError);
```

### 3.4 错误与重连（教学要点）

- `ResourceError` / `DeviceNotFoundError`：设备拔掉后应 `close()`，定时重试或提示用户重新选择端口。  
- 不要在 `readyRead` 里 **同步阻塞** `waitForReadyRead`（会把事件循环卡死）。

## 四、串口接收缓冲与「帧定界」设计（核心）

### 4.1 为什么会出现半包/粘包？

`QSerialPort::readAll()` 每次返回的字节数是**不确定**的：  
可能半帧、一帧、多帧粘在一起。因此必须维护 **持久化接收缓冲** `m_rxBuffer`。

### 4.2 定界算法（与 `parserData` 配合）

在 `m_rxBuffer` 上循环：

1. 在缓冲区中查找同步头 `0xAA 0xAA`（可从索引 0 开始，找不到则丢弃到倒数第二个字节之前）。  
2. 读取第三字节得 `N = PayloadLength`。若 `N==0` 或 `N>170`（与你校验策略一致），**丢弃第一个 `0xAA`，重新对齐**（防止误同步）。  
3. 若 `m_rxBuffer.size() < 4 + N`，**等待更多数据**（`return`，等下次 `readyRead`）。  
4. 否则截取 `QByteArray frame = mid(0, 4+N)`，从 `m_rxBuffer` 移除已消费部分。  
5. 将 `frame` 交给 `parserData`；根据返回值 `-2` 理论上不应再出现（因为已按长度切帧），`-1`/`-3` 可做统计与日志。

### 4.3 伪代码（教学用）

```cpp
void YourClass::onReadyRead()
{
    m_rxBuffer.append(m_port->readAll());

    const int maxGuard = 64 * 1024; // 软上限，防异常洪水
    if (m_rxBuffer.size() > maxGuard) {
        qWarning() << "RX overflow, clearing buffer";
        m_rxBuffer.clear();
        return;
    }

    while (true) {
        int i = m_rxBuffer.indexOf(QByteArrayView("\xAA\xAA"));
        if (i < 0) {
            if (m_rxBuffer.size() > 1)
                m_rxBuffer.remove(0, m_rxBuffer.size() - 1); // 保留最后一个字节可能是第一个AA
            return;
        }
        if (i > 0)
            m_rxBuffer.remove(0, i); // 丢弃同步头之前垃圾

        if (m_rxBuffer.size() < 3)
            return;

        const unsigned char n = static_cast<unsigned char>(m_rxBuffer[2]);
        if (n == 0 || n > 170) {
            m_rxBuffer.remove(0, 1); // 对齐失败，滑动一字节继续找
            continue;
        }

        const int frameLen = 4 + static_cast<int>(n);
        if (m_rxBuffer.size() < frameLen)
            return;

        QByteArray oneFrame = m_rxBuffer.left(frameLen);
        m_rxBuffer.remove(0, frameLen);

        bool isRaw = false, isEeg = false;
        short rawValue = 0;
        eegPkt pkt{};
        int ret = parserData(oneFrame, isRaw, rawValue, isEeg, pkt);
        if (ret == 0) {
            if (isRaw) {
                // 写入高频 Raw 环形缓冲（512Hz）
            }
            if (isEeg) {
                // 写入低频 BigPackage 缓冲（~1Hz）
            }
        } else if (ret == -1 || ret == -3) {
            // 校验失败或长度非法：已切帧仍失败，可记 metrics
        }
        // ret == -2 在已完整切帧时不应出现
    }
}
```

### 4.4 与你描述的「512×8 + 1×36」如何对应？

- **工程上**：不必强行把 TCP/IP 式的「小包=8字节」与物理帧一一对应；关键是 **接收端按 `0xAA 0xAA` 定界**，自然支持任意合法 `N`。  
- **统计上**：你可以每秒统计 `isRaw==true` 次数是否约 512、`isEeg` 是否约 1，用于**健康度监控**（丢包、波特率错误、校验失败率）。

## 五、与 `DataBuffer`（你项目内）的衔接建议

你已有 `DataBuffer`：`appendRawvalue` / `appendBigpackage` 分离高频与低频，这与串口侧 **双速率** 完全同构。

建议约定：

- **`isRaw == true`**：`appendRawvalue({ .raw = rawValue_as_double, .raw_count = ++m_rawCounter, … })`  
- **`isEeg == true` 且需要更新大包**：把 `pkt` 映射到 `BigPackageSample`（8 段 band + signal/att/med），`appendBigpackage(...)`，并记录 `raw_count_at_update` 便于和 RAW 时间对齐。

## 七、常见踩坑（对照你修复清单的「工程版」）

1. **`while (i < size-1)` 类 off-by-one**：定界与 `parserData` 内部循环都要保证**最后一个 checksum 字节可读**。  
2. **把 `readAll()` 结果直接当一帧**：必须先入 **`m_rxBuffer`** 再切帧。  
3. **在高频路径里 `qDebug` 全开**：512Hz 打印会严重拖慢接收；改用**节流日志**。  
4. **RAW 用 `short` 的符号性**：若设备给出的是无符号 16 位，需在绘图前统一转为 `int32` 再减偏置（视硬件定义而定）。  
5. **解析成功但 UI 不更新**：检查是否跨线程直接碰 QWidget；用信号槽 `Qt::QueuedConnection`。

## 八、最小「学习验收」标准

1. 拔掉设备，`errorOccurred` 能提示且不崩溃。  
2. 接上设备，`isRaw` 每秒约 512 次（允许抖动），曲线连续。  
3. 人为暂停 UI 线程 1 秒，接收缓冲**不无限增长**（有上限与策略）。  
4. 校验失败有计数，长时间运行不内存膨胀。
