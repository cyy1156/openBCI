# 串口 `readAll()`、缓冲区与「先二进制还是先十六进制」说明

> 本文**不修改**你工程里的 `device/serialport.cpp` / `serialport.h`，只说明原理并给出**可选改法**（供你自行粘贴或新建类时使用）。

---

## 一、`readAll()` 有没有「缓冲区大小」参数？

**没有。** `QIODevice::readAll()` 的含义是：**把当前可读的字节一次性全部读出来**，返回 `QByteArray`，长度由**此时驱动里已经收到多少字节**决定，每次调用可能不同。

因此你在 `readyRead` 里写的：

```cpp
const QByteArray chunk = m_port.readAll();
```

**不是在规定「每次读多少」**，而是「有多少读多少」。

### 和你配置里的 `readBufferSize` 是什么关系？

你在 `SerialPortConfig` 里有 `readBufferSize`，并在 `applyConfig()` 里调用了：

```cpp
m_port.setReadBufferSize(m_cfg.readBufferSize);
```

这是 **`QSerialPort` 对系统/驱动侧接收缓冲的提示**（建议内核或底层缓冲不要过小），**不是** `readAll()` 的「每次读取长度」，也不会让 `readAll()` 变成固定块大小。

**结论：**

- **单次读到的长度**：由硬件到达节奏 + OS 调度决定，**不固定**。  
- **应用层若要「大缓冲、组帧」**：需要自己维护一个 **`QByteArray m_rxBuffer`**（或环形缓冲），在**别的类**里 `m_rxBuffer.append(chunk)`，与 `SerialPort` 里是否 `readAll` 无关。

## 二、应该先放「原始二进制」还是先转成「十六进制」再进缓冲？

### 推荐做法（脑电/二进制协议通用）

**先把原始二进制放进缓冲区，组帧与解析全部在二进制上做；十六进制只在「给人看」时临时转换。**

| 步骤 | 存什么 | 说明 |
|------|--------|------|
| 串口读出 | `QByteArray chunk` | 每个元素是一个字节 0–255 |
| 组帧缓冲 | `m_rxBuffer.append(chunk)` | 仍是 **二进制** |
| 找同步头 / 长度 / 校验 | 对 `m_rxBuffer` 按下标读 `unsigned char` | 与协议 PDF 一致 |
| 调试窗口 / 日志 | `toHexSpaced(slice)` 得到 `QString` | **仅展示**，不参与解析 |

### 为什么不建议「先转十六进制再进缓冲」？

1. **体积膨胀**：每个字节变成 2 个十六进制字符，若再加空格，约为 **2～3 倍** 以上内存与 CPU。  
2. **协议无法直接解析**：同步头在二进制里是 `0xAA 0xAA`；若变成字符串 `"aa aa"`，你还要再解析字符串才能还原成字节，多此一举且易错。  
3. **`QString` 与编码**：十六进制展示本质是文本；中间若混入 Unicode、大小写、分隔符规则，和「按字节组帧」是两条链路。  
4. **性能**：高频（例如 512Hz 场景）下，每包都做 `toHex` 再拼接，会明显加重 CPU 与日志 IO。

**一句话：二进制是「真数据」，十六进制是「给人看的视图」。缓冲与解析应站在真数据这一侧。**

## 三、和你当前 `SerialPort` 类的分工建议

你现在的设计是：

- `readyRead` → `readAll()` → `emit dataReceived(chunk)`，`chunk` 是 **`QByteArray`**。

这已经符合「**线上始终是二进制**」。

**建议保持：**

- `SerialPort` **不要**在类内部把数据改成十六进制再发信号；  
- 继续发 `dataReceived(const QByteArray &chunk)`；  
- 新建例如 `EegRxAssembler`（名字随意）：

  ```text
  connect(serial, &SerialPort::dataReceived, assembler, &EegRxAssembler::onBytes);
  ```

  在 `onBytes` 里：`m_rxBuffer.append(chunk);` 然后 `while` 切完整帧。

**若要在 UI 上显示 HEX：** 只对**某一小段**（例如最近 64 字节）调用 `SerialPort::toHexSpaced(...)`，并**节流**（例如每 100ms 更新一次），不要对每个 `chunk` 无限制 `qDebug`。

## 四、若你要「规定应用层缓冲上限」（可选改法示例）

以下代码**仅作文档示例**，用于「组帧类」而非强制改 `SerialPort`：

```cpp
// 示例：组帧类成员
QByteArray m_rxBuffer;
static constexpr int kMaxRxBytes = 64 * 1024;

void EegRxAssembler::onBytes(const QByteArray &chunk)
{
    m_rxBuffer.append(chunk);
    if (m_rxBuffer.size() > kMaxRxBytes) {
        // 策略任选：清空 / 只保留尾部 / 打日志告警
        m_rxBuffer.clear();
    }
    // ... 在此 while 提取完整帧 ...
}
```

`kMaxRxBytes` 的取值可参考 `docs/脑电串口解析与缓冲设计教学.md` 里的流量估算。

## 五、小结（直接回答你的判断题）

| 问题 | 建议 |
|------|------|
| `readAll()` 要不要规定「读多少字节」？ | **不用也不能**；它读当前可读的全部。需要「块大小」时用应用层缓冲自己攒。 |
| 先二进制进缓冲再转十六进制，还是先十六进制再进缓冲？ | **先二进制进缓冲**；十六进制仅在显示/调试时从二进制**临时**生成。 |
