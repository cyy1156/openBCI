# 后期详细方案（低刷新 + 采集不丢包）— 1 秒微计划

> 目标：在尽量降低 UI 刷新次数/重绘频率的前提下，保证采集链路实时、尽量不丢包；同时把“所有数据”保存为 CSV/HEX（绝对记录），并允许绘图丢弃旧数据以维持实时体验。
>
> 本文是工程设计方案文档，不直接改你的源码；你可以照着把类拆出来、把连接关系和线程模型搭起来。

---

## 0. 总体架构（推荐的“管线职责分离”）

把系统拆成 4 层，每层职责单一：

1. **串口字节层（IO）**：`SerialPort` 只负责 `readyRead -> readAll -> emit dataReceived(chunk)`  
2. **组帧层（协议边界）**：`ThinkGearFrameAssembler` 用 `m_rxBuffer` 按 `AA AA + N + payload + checksum` 切出整帧  
3. **帧内解析层（TLV 字段）**：`ThinkGearPayloadParser` 从 Payload 解析出
   - `rawReceived(qint16)`（高频）
   - `signalQuality/attention/meditation/blink`（低频）
   - `eegPowerReceived(...)`（低频）
4. **业务层（保存、算法、绘图）**：拆成
   - **记录层（CSV/HEX 落盘）**：高优先级、保证“不丢记录”
   - **算法层（uV + 滤波 + FFT + 能量）**：按窗口处理
   - **绘图层（低刷新）**：从缓冲拿最新窗口，旧数据可丢弃

关键原则：
- **不丢包**优先保证的是：串口/组帧/解析链路要“快”，不要被写盘和重计算阻塞。
- **允许绘图丢旧数据**：因为 UI 追求的是实时展示，不是 100% 离线复现。

## 1. “尽量降低刷新次数”怎么做才不影响不丢包？

### 1.1 UI 刷新 vs 数据采集是两件事
- `readyRead` / 组帧 / 解析链路不依赖 UI 刷新频率。
- UI 只负责：把“最新可用的缓存窗口”画出来。

### 1.2 降低刷新频率的建议值（可直接用）
- **绘图刷新**：`30~60 ms` 一次（约 16~33 FPS）。  
  若你显示的是波形趋势，一般 `25~50 FPS` 已足够稳定。
- **HEX 文本/调试输出**：默认关闭或极低频，例如每 `1~2 秒` 打一次摘要，不要把每个 chunk 都 `qDebug`。
- **CSV flush（真正落盘）**：
  - 绝对不丢记录：尽量设置为“批量 + 定期 flush”（例如每 `50~200 ms` flush 一次，或累积到 `几百行` 再 flush）。
  - 如果磁盘慢，flush 太频繁会拖慢记录线程，进而间接影响队列。

## 2. CSV/HEX“绝对实时保存且不拖采集”方案

你的需求是“所有数据都保存”，因此不能直接在采集链路里写文件。

### 2.1 建议使用：有界队列（Bounded Queue）+ 记录线程（Writer Thread）

**记录队列**建议设计两类队列（或一个带结构体的队列）：

1. **`hexFrameQueue`**：写 HEX（整帧）
2. **`rawMetricsQueue`**：写解析后的数值（uV、注意力、能量等）

队列必须：
- **有上限**（防止算法/磁盘异常时内存无限增长）
- 满了怎么办要明确策略（见下文）

### 2.2 满队列策略（决定“是否丢数据”）

你说“绝对不想丢”，那就不要默认“丢旧数据”策略；更合理的是：
- 队列足够大（按你估计的最坏磁盘延迟扩容），尽量避免满队列
- 写盘线程尽快落盘，flush 批量
- 若仍满：可以选择阻塞生产者（可能导致串口缓冲顶满从而丢包）  
  所以建议的折中是：
  - 记录队列设得足够大（比如能扛住 2~5 秒写盘卡顿）
  - 队列满时：仅对**绘图**丢旧，不对**记录**丢数据

### 2.3 HEX/CVS 的字段建议（便于后期复现）

1. **HEX 绝对记录文件**（用于排查协议/校验）
   - `timestamp_ms, frame_index, frame_hex`
   - 可选：`frame_len`
2. **数值 CSV 文件**（用于算法与可视化）
   - `timestamp_ms, raw_uV, attention, meditation, signal_quality, blink`
   - EEG 功率若不是每帧都有，可以单独一张文件：
     - `timestamp_ms, delta, theta, lowAlpha, highAlpha, lowBeta, highBeta, lowGamma, midGamma`

### 2.4 写盘批量策略（减少刷新次数的关键）

- 记录线程每次从队列 pop 一批（比如 50~500 条），拼接成字符串/行缓存，然后一次性写入文件。
- 每 `100 ms` 或当缓存超过一定大小再 `flush()`。

这样能兼顾：
- 写盘开销低
- 采集链路不被阻塞
- 记录接近实时

## 3. RAW -> 微伏（uV）-> 算法（你后期的完整步骤落地）

### 3.1 uV 转换公式（按你的 NeuroSky 思路）

你给出的换算（电压到微伏）可抽成一个函数：

- 先把串口解析出的 `rawValue` 作为 **16 位有符号整数（int16）**
- 然后：

`voltage_µV = rawValue * (1.8 / 4096) / 2000 * 1,000,000`

数值简化：
`voltage_µV ≈ rawValue * 0.2197`

建议工程化做法：
- 把 `0.2197` 定义为常量 `constexpr double kUvScale = ...`
- 若设备增益有偏差，允许配置 `gain_adjust`（比如 ±5%）

### 3.2 算法管线（建议分两阶段）

阶段 A：预处理与窗口准备（流式）
- 去直流（DC removal）
- 滤波（带通/陷波按你模型要求）
- 加窗（Hann/Blackman 等）
- FFT

阶段 B：频带能量 + 归一化（输出特征）
- 按区间积分能量：
  - delta/theta/alpha/beta/gamma（按你的 band 定义）
- 得到各波段能量向量
- 归一化：
  - `relative = energy / sum(energy)` 或用对数压缩（看你“情感识别”的需求）

### 3.3 什么时候做 FFT？

FFT 必须要“凑够窗口”，所以建议：
- `window_size = 512`（如果你采样率 512Hz，则 1 秒一窗）
- `hop_size = 128~256`（窗口滑动步长，越小越实时但计算越多）

这会影响 CPU：
- 采集链路不做 FFT
- FFT 在算法线程做

## 4. 绘图的数据怎么取（允许丢旧数据）

你的 DataBuffer 已经是环形缓存，天然适合做“只显示最新窗口”：
- 高频 RAW：用 `window_RawSample()` 获取一段最近数据
- 大包 EEG：用 `window_BigPackageSample()` 获取最近一次或最近几次更新

**实时显示只读，不阻塞采集：**
- 绘图刷新定时器触发
- 从 DataBuffer 读窗口（读是拷贝，避免计算时被覆盖）
- 更新曲线

如果你遇到 UI 过慢：
- 直接降低刷新频率（例如从 16ms -> 33ms）
- 或者只更新关键曲线（例如每次只更新 RAW 一条、EEG 每 200ms 更新一次）

## 5. 线程模型（推荐落地做法）

你现在是多线程目标，推荐“最小可落地”：

### 5.1 最简版本（够用且容易不丢包）
- UI 主线程：只负责连接、按钮、绘图定时器
- 串口/解析链路：保持在同一线程（或串口线程）
- 写盘：单独 writer thread
- 算法：单独 analysis thread（可选先不加，看你 CPU）

### 5.2 进阶版本（更稳）
- reader thread（纯读 + append）
- assembler thread（切帧）
- parser + analysis thread（解析 + FFT/特征）
- writer thread（批量写 CSV/HEX）
- UI thread（只读 buffer + 低刷新）

关键是：**队列要有界 + writer batch + UI 定时器低频**。

## 8. 代入参数后的“队列容量估算 + 写盘批量策略”

### 8.1 数据量的量级（用于估算队列）
在常见 ThinkGear 流里，你的业务表现通常是：
- RAW：约 **512 条/秒**
- EEG“大包字段”：约 **1 条/秒**（signal/attention/meditation + 8 bands）
- 帧的数量可能略有不同（因为设备可能把 RAW 和 EEG 放在不同的 payload/帧里），但对“写盘队列”估算而言，**raw 的 512Hz 是主导项**。

### 8.2 写盘线程的节奏（确保不拖采集）
目标：`flush` 延迟 <= 100ms。

建议写盘线程每次 flush 采用“时间驱动 + 批量驱动”：
- **时间驱动**：每 100ms 最多 flush 一次
- **批量驱动**：缓存累计到一定行数（例如 100~300 行）就写入（不一定每次都 flush）

经验上（SSD + 解析后是结构化数值），这样可以把写盘开销压到很低，同时避免队列堆积。

### 8.3 有界队列容量（Bounded Queue）推荐值
队列容量至少要能承受“最坏情况：writer 在 flush 周期内稍慢一点”，例如积压 200ms。

计算（以 512Hz 为主）：
- raw 积压 200ms：`512 * 0.2 ≈ 102` 条

建议给 2~5 倍安全余量，方便应对操作系统调度抖动：
- **rawMetricsQueue 容量**：建议 **512 或 1024**
- **hexFrameQueue 容量**：建议 **512 或 1024**（写 HEX 更慢时宁可稍大）

如果你后续发现队列偶尔满：
- 优先不要丢“记录队列”（你说了不想丢记录）
- 你可以选择“写盘线程更激进”（更频繁批量写但仍不每行 flush）

## 9. CSV/HEX 文件建议（既全量又不拖采集）

### 9.1 不建议把所有内容都混在一个大 CSV
原因：HEX 字符串比数值字段更重，混在一个文件里会让写盘耗时波动更大，从而间接影响采集侧“是否来得及消费”。

建议分两类文件：
1. **HEX 绝对记录（可选按帧）**：`raw_frames_hex.csv`
   - 字段：`timestamp_ms, frame_index, frame_hex`
   - 其中 `frame_hex = frame.toHex(' ')`（或去空格版本看你习惯）
2. **数值 CSV（算法/绘图用）**：`metrics.csv`
   - RAW 行（512Hz）：  
     `timestamp_ms, raw_int16, raw_uV`
   - EEG“大包行”（约 1Hz）：  
     `timestamp_ms, signal_quality, attention, meditation, delta, theta, lowAlpha, highAlpha, lowBeta, highBeta, lowGamma, midGamma`

这样做的好处是：你后期做 FFT/特征时只读取 `metrics.csv`（数值更轻更快），HEX 文件只用于追查协议问题。

### 9.2 实时“绝对记录”的关键点（总结）
- 写盘只在 **writer thread** 做
- 采集/解析线程只把数据 `push` 到有界队列
- writer 每次 `pop` 后批量拼接，然后一次性写入
- `flush` 控制在 <= 100ms（或累计到足够行数再 flush）

## 10. FFT/能量计算建议（FFT 1 秒窗口）

你给定 FFT 窗口 = 1 秒：
- `window_size = 512`（与 RAW 512Hz 对齐）

建议 hop_size（二选一）：
- **0.5 秒更新**：`hop_size = 256`（实时性不错、计算量适中）
- **0.25 秒更新**：`hop_size = 128`（更平滑更实时，但计算更多）

频带能量输出：
- 对每个窗做 FFT -> 映射频率 bin -> 在 delta/theta/alpha/beta/gamma 区间积分能量
- 再做归一化：`relative = energy / sum(energy)` 或 `log(energy + eps)`（看你情感识别模型输入要求）
